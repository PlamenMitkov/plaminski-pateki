8. Synthetic / Test Data Assessment

Important distinction:
- The structural trail records are not synthetic in the usual sense; they come from the project dataset and its import/scrape process.
- The assistant-enriched narrative and summary fields are derived text. They are synthetic or semi-synthetic outputs, even when grounded in real sources.

Practical classification:
- Non-synthetic / base data: ids, names, coordinates, region, route metadata, source links already present in the imported dataset.
- Derived synthetic-like data: rewritten descriptions, summaries, highlights, normalized terrain descriptions, reviewer notes, confidence labels, and some normalized user-facing fields.

Recommended usage:
- Good for QA, UI testing, search testing, ranking tests, offline package tests, demo content, and pilot product content.
- Acceptable for production display if clearly treated as editorial/assistive copy rather than official authority.
- Not ideal as a strict source of record for legal, safety-critical, or official-tourism claims without human review.

11. Recommended Next Architecture Improvements

11.1 schema.org / structured data

Recommendation:
- publish a JSON-LD block for each trail page
- use schema.org TouristAttraction as the baseline public type for discoverability
- attach geo coordinates via GeoCoordinates
- expose canonical source URLs through sameAs or url where appropriate
- map existing dataset fields into structured properties where possible

Practical field mapping recommendation:
- name -> name
- description -> description
- coordinates -> geo.latitude and geo.longitude
- photo_url -> image or photo
- location.keywords -> keywords
- source or metadata.sources_urls -> sameAs or url
- accessibility / public visitor information -> publicAccess and isAccessibleForFree when known
- map links -> hasMap

Important modeling note:
- schema.org does not give a perfect hiking-trail domain model for every trail-specific property already present in eco.json
- for route-specific details such as difficulty, estimated duration, terrain notes, elevation gain, and seasonal cautions, use additionalProperty or a separate internal API schema instead of forcing a misleading public structured-data model

Recommended purpose of schema.org here:
- SEO and discoverability
- better interoperability with search engines and tourism platforms
- clean public-facing canonical metadata layer, separate from the richer internal application schema


11.2 API security recommendations

For the current project, the highest-value recommendations are:

1. Require HTTPS everywhere.
2. Enforce authorization checks per endpoint and per trail/resource id.
3. Keep JWT validation strict: verify issuer, audience, expiry, and signature algorithm from server config, not from token header trust.
4. Add strict input validation for ids, uploaded files, filters, paging, and any enrichment/admin endpoints.
5. Use content-type allowlists and reject unsupported media types.
6. Apply rate limiting and return 429 for abuse-prone endpoints, especially assistant/chat, enrichment, image upload, and external API proxy endpoints.
7. Do not place API keys or secrets in URLs, logs, or client-visible payloads.
8. Protect management/admin endpoints separately and avoid exposing them broadly.
9. Log security-relevant events and validation failures without leaking secrets.
10. Treat third-party API responses as untrusted input.

Most relevant OWASP API risk areas for this project:
- broken object level authorization
- broken function level authorization
- unrestricted resource consumption
- server-side request forgery
- unsafe consumption of APIs
- improper inventory management


11.3 Dynamic data from external APIs

Recommendation:
- keep eco.json as the canonical base dataset
- layer dynamic fields from external APIs at runtime or through short-lived cache, instead of hard-writing them permanently into the core dataset unless reviewed

Good candidates for dynamic enrichment:
- weather forecast
- route time estimates
- live transport context
- temporary closures or access warnings
- wildfire, flood, or hazard alerts
- seasonal public-opening status

Recommended design pattern:
- static layer: eco.json and reviewed assistant-enriched metadata
- cached dynamic layer: external API responses with explicit TTL
- merged response layer: API combines static data plus dynamic data on read

Recommended metadata to store for every dynamic field:
- source provider
- fetched_at timestamp
- expires_at timestamp
- confidence/trust level
- fallback policy if upstream is unavailable

Security recommendations for external APIs:
- allowlist target domains
- set strict timeouts and retries with backoff
- validate response schema before use
- isolate provider failures so they do not break base trail pages
- do not trust third-party fields more than local validated data
- store secrets only in server-side configuration


11.4 Data governance recommendation

Use three trust levels in the project:
- canonical: reviewed static fields in eco.json
- derived/editorial: assistant-generated descriptions and summaries
- dynamic/external: cached API overlays with timestamps and provenance

That separation will keep product behavior clear, safer, and easier to audit.
